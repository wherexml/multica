# Multica 当前仓库 UI 设计规范（design.md）

## 1. 文档定位

本文件不是产品蓝图，也不是需求列表，而是**基于当前仓库已存在 UI 结构、组件模式和页面壳层**整理出的设计规范。  
后续所有设计与前端改造，均应优先遵从本规范，再叠加供应链 Demo 的业务语义。

适用范围：

- `apps/web/app/(dashboard)/*`
- `packages/views/*`
- `packages/ui/*`
- 所有新增的业务页面、抽屉、Tab、卡片、表格、状态标签、表单与空状态

---

## 2. 现有 UI 基线

### 2.1 应用壳层

当前 Web 端采用 App Router，主业务壳在 `apps/web/app/(dashboard)/layout.tsx`，其核心不是“空白页面”，而是一个已经成型的协作控制台：

- 外层使用 `DashboardLayout`
- 顶部包含 `SearchTrigger` / `SearchCommand`
- 页面中长期存在 `ChatWindow` / `ChatFab`
- 品牌入口使用 `MulticaIcon`

这意味着后续设计必须默认存在：

1. 左侧侧边栏
2. 顶部检索入口
3. 右下角聊天/辅助入口
4. 中央内容区
5. 移动端 `SidebarTrigger`

**设计约束**  
后续新增页面不得绕开这套壳层另造一套全屏业务后台，除非明确进入独立流程页。

### 2.2 侧边栏结构

当前 `packages/views/layout/app-sidebar.tsx` 已经形成固定导航模式：

- Personal 区：`Inbox`、`My Issues`
- Workspace 区：`Issues`、`Projects`、`Agents`
- Configure 区：`Runtimes`、`Skills`、`Settings`
- 还包含 `Pinned` 区域
- `Inbox` 有未读数
- 支持 pin / create issue 快捷入口

**设计约束**

- 保留三段式导航组织方式，不把所有业务入口打散成十几个一级菜单
- 业务语义替换应优先发生在 `labelKey` 和页面内容层，不先推翻导航骨架
- `Pinned` 继续保留，作为高频对象入口，而不是改成独立收藏页

### 2.3 页面布局模式

当前页面已形成三种清晰的结构原型：

#### A. 主从结构
`InboxPage` 使用 `ResizablePanelGroup` 做左列表、右详情的主从布局。  
适用于：

- 工作台 / 告警中心
- 审批中心
- 连接器中心
- 运行记录中心

#### B. 列表 / 看板切换结构
`IssuesPage`、`ProjectDetail` 已具备：

- `IssuesHeader`
- `BoardView`
- `ListView`
- `BatchActionToolbar`

适用于：

- 决策单中心
- 专题下决策单
- 审批列表
- 仿真任务列表

#### C. 详情协作结构
`IssueDetail` 已具备非常成熟的详情壳：

- `TitleEditor`
- `ContentEditor`
- 元信息区
- `Activity`
- `Sub-issues`
- Due date / Pin / Copy link / Create sub-issue 等操作

适用于：

- 决策单详情
- 审批详情
- 动作执行详情
- 重要连接器详情

**设计约束**

- 后续业务设计应“借壳换语义”，不应轻易替换页面骨架
- 复杂业务信息优先通过 Tab、分区卡片、摘要区、右侧附加面板叠加
- 不建议为每个新对象都重做一套详情页交互模型

### 2.4 设置页模式

当前 `SettingsPage` 使用纵向 `Tabs`，并已形成“左侧导航 + 右侧配置面板”的结构。  
这意味着新增：

- 语言
- 连接器
- 审批
- 审计
- 模型管理

时，优先作为设置页扩展 Tab 或子分组接入，而不是另做完全独立的管理后台。

### 2.5 视觉令牌与样式基线

当前样式基线在 `packages/ui/styles/tokens.css` 与 `base.css` 中，已经定义：

- light / dark 双主题颜色令牌
- `sidebar-*` 专用颜色令牌
- radius 层级
- 全局 `font-sans`
- `bg-background` / `text-foreground`
- 滚动条与弹出层状态样式

**设计约束**

- 所有新增界面优先复用设计令牌，不新增“业务色板体系”
- 禁止在业务组件中直接写大量硬编码颜色、圆角、阴影、间距
- 深色模式必须被视为默认兼容范围，而不是后补

## 3. 总体设计原则

### 3.1 先复用结构，再替换语义
当前仓库最有价值的不是“业务名词”，而是已经打磨过的协作结构。  
供应链 Demo 的设计应该：

- 保留页面骨架
- 替换文案
- 增加业务字段
- 增加业务分区
- 增加供应链对象关系

而不是直接推翻成另一套 SaaS 后台。

### 3.2 中文优先，但不破坏现有 locale 机制
当前仓库已经存在 `LocaleSync`，但根布局默认 `lang="en"`，且同步逻辑偏轻量。  
设计与实现必须遵循：

- 默认展示 `zh-CN`
- 兼容 `en-US`
- 保留当前“不要在 root layout 里直接依赖 `cookies()`”的思路
- `multica-locale` 需要支持 `zh-CN / en-US`，同时兼容现存 `zh / en`

### 3.3 信息密度优先于大面积装饰
当前 UI 更接近“高密度协作控制台”，不是营销站。  
后续供应链页面也应保持：

- 表格优先
- 结构清楚
- 状态明确
- 少大图、少无效插画、少情绪化装饰
- 重点放在筛选、状态、解释、操作链路

### 3.4 所有新增设计都要可映射到现有组件
新增页面优先复用：

- `Sidebar`
- `Tabs`
- `Popover`
- `DropdownMenu`
- `ResizablePanelGroup`
- `BoardView`
- `ListView`
- `Button`
- `Badge`
- `Editor`
- `Command/Search`

任何新的视觉稿都应先回答：是否能在现有组件栈中落地。

## 4. 导航与信息架构规范

## 4.1 导航语义映射

| 当前导航 | 对外交付语义 | 路径建议 |
|---|---|---|
| Inbox | 工作台 / 待办与告警 | 保留 `/inbox` |
| My Issues | 我的待办 | 保留 `/my-issues` |
| Issues | 决策单中心 | 保留 `/issues` |
| Projects | 专题中心 | 保留 `/projects` |
| Agents | 专家 Agent | 保留 `/agents` |
| Runtimes | 执行环境 | 保留 `/runtimes` |
| Skills | 技能包 | 保留 `/skills` |
| Settings | 平台设置 | 保留 `/settings` |

### 4.2 一级导航新增原则

仅在现有入口无法承载时新增一级路由。  
本期允许新增但应谨慎控制的入口：

- `/tower`：控制塔 / 业务透镜
- `/approvals`：审批中心
- `/connectors`：连接器中心

优先级顺序：

1. 先扩展已有页面
2. 再考虑设置页内嵌子项
3. 最后才新增一级路由

### 4.3 Pinned 规范

Pinned 是当前 UI 的高频入口机制。  
后续设计中：

- 决策单
- 专题
- 连接器
- 关键 Agent
- 常用执行环境

都允许被 pin 到侧边栏。  
不要另做“收藏中心”替代现有 pin 机制。

## 5. 页面级设计规范

## 5.1 工作台（基于 InboxPage）

### 当前可复用结构
`InboxPage` 现有能力：

- 左侧列表
- 右侧详情
- 可拖拽宽度
- 空状态
- 批量已读 / 归档操作

### 目标设计
工作台应继续沿用这套主从结构，但内容切换为：

#### 左侧列表
- 异常告警
- 待审批
- 待处理决策单
- 运行中决策

#### 右侧详情
- 触发原因
- 关键指标变化
- 关联对象
- 推荐动作
- 一键生成/打开决策单
- 最近执行轨迹

### 设计规则
- 左侧永远是“可扫读”的高密度列表
- 右侧永远是“当前条目的上下文详情”
- 不把工作台改成纯卡片瀑布流
- 不把工作台改成聊天首屏

## 5.2 决策单中心（基于 IssuesPage）

### 当前可复用结构
`IssuesPage` 已具备：

- Board / List 切换
- `IssuesHeader`
- `BatchActionToolbar`
- 过滤与显示设置入口

### 目标设计
继续用现有列表/看板骨架，但补充供应链语义字段：

- 决策类型
- 业务对象
- 风险等级
- 决策阶段
- 执行模式
- 所属专题
- 负责人
- 更新时间

### 设计规则
- Board 视图继续用于阶段流转
- List 视图用于高密度筛选与批量操作
- `IssuesHeader` 扩展，不重做顶栏
- 所有新筛选项都收口在现有 Header 过滤体系内

### 建议新增筛选维度
- `phase`
- `risk_level`
- `execution_mode`
- `decision_type`
- `project_id`
- `warehouse`
- `region`
- `sku`

## 5.3 决策单详情（基于 IssueDetail）

### 当前可复用结构
`IssueDetail` 已有成熟的协作能力：

- 标题编辑
- 描述编辑
- 元数据编辑
- Activity
- 子任务
- Pin / Copy link
- due date
- 操作菜单

### 目标设计
不重做整个详情页，而是在现有详情壳上升级为“业务详情 + 协作详情”的双层结构。

### 推荐结构
```text
DecisionDetailShell
  ├─ Header（复用 IssueDetail 的标题与主操作）
  ├─ Summary Strip（新增）
  │    ├─ 决策阶段
  │    ├─ 风险等级
  │    ├─ 执行模式
  │    ├─ 业务对象
  │    └─ 所属专题
  ├─ Main Tabs（新增）
  │    ├─ 概览
  │    ├─ 指标快照
  │    ├─ 诊断分析
  │    ├─ 场景仿真
  │    ├─ 推荐方案
  │    ├─ 审批流
  │    ├─ 动作执行
  │    └─ 协同记录 / 执行轨迹
  ├─ Existing Activity / Comments（保留）
  └─ Sub-issues（保留，作为诊断/仿真/执行子任务）
```

### 关键约束
- 标题、描述、Assignee、状态、优先级、Due date 继续保留
- Tab 是业务内容层；Activity / Sub-issues 仍然是协作底座
- 不建议把所有内容堆到单页长滚动中
- 不建议删除已有 `Create sub-issue`、`Pin/Unpin`、`Copy link`

### 子任务设计规范
子任务用于承载步骤性工作，而不是复制主决策内容。  
建议子任务类型：

- 拉数
- 诊断
- 仿真
- 核价
- 审批准备
- 执行确认
- 复盘

## 5.4 专题中心（基于 Projects）

### 当前可复用结构
`ProjectDetail` 已经复用 `IssuesHeader + BoardView + ListView + BatchActionToolbar`，并带有进度摘要。

### 目标设计
专题中心应继续使用“专题摘要 + 下属决策列表”的结构。

### 顶部建议增加
- 专题目标
- 关键指标
- 风险摘要
- 决策阶段分布
- 最近执行结果

### 设计规则
- 专题不是独立的 BI 大屏
- 专题详情的主体仍是“该专题下的决策单流”
- 板块和列表视图继续复用 Issues 系组件，避免双套交互

## 5.5 专家 Agent / 执行环境 / 技能包

### 专家 Agent
以现有 `Agents` 页面为壳，新增：

- 能力域
- 输入要求
- 绑定技能包
- 可用执行环境
- 最近成功率
- 适用场景

### 执行环境
以现有 `Runtimes` 页面为壳，新增：

- `executor_kind`
- 网络区域
- 凭证作用域
- 资源限额
- 允许动作
- 是否需要审批

### 技能包
以现有 `Skills` 页面为壳，新增：

- 版本
- 输入/输出 schema
- 适用业务域
- 依赖执行环境
- 依赖连接器
- 评测结果
- 回滚说明

## 5.6 平台设置（基于 SettingsPage）

### 当前可复用结构
`SettingsPage` 已经是纵向 Tabs 模式。

### 后续扩展原则
新增能力优先挂到设置体系中：

- 语言
- 连接器
- 审批规则
- 审计
- 模型与评测
- 工作空间默认值

### 设计规则
- 设置页是“配置中心”，不是二次产品首页
- 复杂表单优先放右侧内容区
- 需要列表+详情的配置项，可在 Tab 内继续使用主从结构

## 6. 交互规范

## 6.1 Header 与操作区

页面头部遵循统一层次：

1. 页面名称
2. 页面说明（必要时）
3. 一级筛选或范围切换
4. 主要动作按钮
5. 次级动作置于 Dropdown / More 菜单

不要在每个业务页重新发明一套标题区。

## 6.2 状态展示

供应链 Demo 中会出现大量状态。设计上必须区分：

- 协作状态：沿用 issue status
- 决策阶段：`identified / diagnosing / ...`
- 审批状态：`draft / pending / approved / rejected`
- 执行状态：`pending / running / succeeded / failed / rolled_back`
- 风险等级：`low / medium / high / critical`

**规则**

- 不同状态维度不可混为一个 Badge
- 列表中优先显示“阶段 + 风险”
- 详情中显示完整状态组

## 6.3 列表与表格

所有高频运营页面优先表格或高密度列表。  
表格列的设计原则：

- 第一列永远是主对象名
- 第二层列放业务关键信息
- 风险/状态/执行模式使用短标签
- 尽量保留批量操作能力

## 6.4 抽屉与弹层

适合用抽屉的场景：

- 从告警快速生成决策单
- 审批意见填写
- 连接器测试连接
- 执行动作确认

适合用详情页的场景：

- 决策单全量上下文
- 专题详情
- 重要 Agent / Runtime / Skill 详情

原则：  
临时动作用抽屉；长期对象用详情页。

## 6.5 搜索与命令面板

当前应用已内置 `SearchTrigger` 与 `SearchCommand`。  
后续设计必须把以下对象纳入搜索结果：

- 决策单
- 专题
- Agent
- Runtime
- Skill
- 连接器
- 审批项

不要为每个模块单独再做一套搜索入口。

## 7. 文案与 i18n 规范

## 7.1 术语固定映射

| 英文内核术语 | 中文展示 |
|---|---|
| Workspace | 业务空间 |
| Project | 项目 / 专题 |
| Issue | 决策单 |
| Inbox | 工作台 / 待办与告警 |
| My Issues | 我的待办 |
| Agent | 专家 Agent |
| Runtime | 执行环境 |
| Skill | 技能包 |
| Comment | 协同记录 |
| Activity / Run messages | 执行轨迹 / 协同记录 |

### 7.2 文案规则
- 中文模式下，尽量不直接向终端用户暴露 `Issue`、`Workspace`、`Runtime daemon` 等底层协作术语
- 少量技术术语保留英文：Agent、Runtime、Skill、API、SQL、LLM
- 同一对象全站只保留一个主中文名称
- 空状态、错误提示、表单帮助文案都必须进消息包

### 7.3 当前实现兼容要求
由于当前仓库已存在轻量 locale scaffold，后续实现必须兼容以下事实：

- 根布局当前默认 `lang="en"`
- `LocaleSync` 是现有语言同步入口
- cookie 名为 `multica-locale`
- 现有逻辑偏向 `zh`，未来需扩展为 `zh-CN / en-US`

因此设计稿与开发说明中必须明确：

1. 语言切换不要通过重构整站路由实现
2. 优先升级现有 locale 机制
3. 允许旧 cookie 值继续可读，再规范化写回新值

## 8. 视觉与组件规范

## 8.1 颜色
- 使用 `tokens.css` 里的背景、前景、边框、侧边栏变量
- 风险等级和状态颜色应基于现有 token 系统做最小扩展
- 不新增独立“供应链品牌色体系”

## 8.2 间距
- 优先沿用现有卡片、列表、表格、Panel 的间距尺度
- 新页面不要出现与现有页面明显不同的 padding 节奏
- 不要通过过度留白伪装“高级感”

## 8.3 组件复用优先级
1. 复用 `packages/ui` 现有组件
2. 基于现有组件做轻量包装
3. 最后才新增基础组件

## 8.4 深色模式
- 新页面默认要检查 dark mode
- 不能只在浅色主题下保证可读性
- 图表、标签、边框色也需要在 dark mode 下可用

## 9. 数据绑定与 API 使用规范

### 9.1 列表页
列表页可继续采用资源型 API，如：

- `/api/issues`
- `/api/projects`
- `/api/agents`
- `/api/runtimes`
- `/api/skills`

同时逐步加上供应链字段。

### 9.2 详情页
复杂详情页应优先使用聚合接口，如：

- `/api/decisions/{id}`

原因：

- 当前详情页已经复杂
- 八个 Tab 不适合首屏并发拼多个离散接口
- 聚合接口更符合现有详情壳的渲染模式

### 9.3 设计与开发协同边界
设计稿中若出现以下内容，必须同步指明来源与回写方式：

- 阶段
- 风险等级
- 执行模式
- 审批状态
- 执行状态
- 快照时间
- 推荐方案版本
- 外部单号

避免出现“界面有字段、接口未定义”的设计空转。

## 10. 设计评审检查单

在提交任何新设计前，至少检查以下问题：

### 结构一致性
- 是否沿用了 `DashboardLayout`
- 是否沿用了现有侧边栏分组逻辑
- 是否复用了现有主从 / 列表看板 / 详情协作三种结构之一

### 组件一致性
- 是否优先使用现有 `packages/ui`
- 是否有不必要的新交互控件
- 是否保留了现有 Pin、Search、Batch Action、Tabs 等能力

### 语言一致性
- 是否所有业务文案进入消息包
- 是否术语与全站保持一致
- 是否兼容 `multica-locale` 当前实现路径

### 业务闭环
- 页面是否能支撑“异常 → 决策 → 审批/执行 → 复盘”
- 是否能看到关键信息、关键状态、关键操作
- 是否避免了只能“看”不能“做”的假页面

### 工程可落地性
- 是否明确基于哪个现有页面或组件演进
- 是否避免新增大规模重复组件
- 是否考虑 dark mode、空状态、加载态、错误态

---

## 11. 完成定义

一个设计方案只有在同时满足以下条件时，才视为符合本规范：

1. 能明确映射到当前仓库现有 UI 壳层
2. 不破坏侧边栏、搜索、聊天、Pin 等已存在主交互
3. 中文语义清晰且术语一致
4. 能用现有设计令牌和组件体系落地
5. 对供应链业务是“叠加语义”，不是“推翻底座”
6. 支撑后续开发用最小改造成本完成演示闭环
